## Index Patterns

### Add Indexes on WHERE and JOIN Columns
```sql
-- BAD: No index on foreign key
CREATE TABLE orders (id bigint PRIMARY KEY, customer_id bigint REFERENCES customers(id));

-- GOOD: Index on foreign key
CREATE INDEX orders_customer_id_idx ON orders (customer_id);
```

### Choose the Right Index Type
| Type | Use Case | Operators |
|------|----------|-----------|
| B-tree (default) | Equality, range | `=`, `<`, `>`, `BETWEEN`, `IN` |
| GIN | Arrays, JSONB, full-text | `@>`, `?`, `@@` |
| BRIN | Large time-series tables | Range queries on sorted data |

### Composite Indexes for Multi-Column Queries
```sql
-- BAD: Separate indexes
CREATE INDEX orders_status_idx ON orders (status);
CREATE INDEX orders_created_idx ON orders (created_at);

-- GOOD: Composite (equality columns first, then range)
CREATE INDEX orders_status_created_idx ON orders (status, created_at);
```

## Schema Design Patterns

### Data Type Selection
```sql
-- BAD: Poor type choices
CREATE TABLE users (id int, email varchar(255), created_at timestamp, is_active varchar(5));

-- GOOD: Proper types
CREATE TABLE users (id bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY, email text NOT NULL, created_at timestamptz DEFAULT now(), is_active boolean DEFAULT true);
```

## Security & Row Level Security

### Enable RLS for Multi-Tenant Data
```sql
ALTER TABLE orders ENABLE ROW LEVEL SECURITY;
ALTER TABLE orders FORCE ROW LEVEL SECURITY;

CREATE POLICY orders_user_policy ON orders
  FOR ALL
  USING (user_id = current_setting('app.current_user_id')::bigint);
```

### Optimize RLS Policies
```sql
-- BAD: Function called per row
CREATE POLICY orders_policy ON orders USING (auth.uid() = user_id);

-- GOOD: Wrap in SELECT (cached, called once) — 100x faster
CREATE POLICY orders_policy ON orders USING ((SELECT auth.uid()) = user_id);

-- Always index RLS policy columns
CREATE INDEX orders_user_id_idx ON orders (user_id);
```

## Concurrency & Locking

### Keep Transactions Short
```sql
-- BAD: Lock held during external API call
BEGIN; SELECT * FROM orders WHERE id = 1 FOR UPDATE; -- HTTP call... UPDATE ... COMMIT;

-- GOOD: Minimal lock duration
BEGIN; UPDATE orders SET status = 'paid' WHERE id = $2 AND status = 'pending' RETURNING *; COMMIT;
```

### Use SKIP LOCKED for Queues
```sql
UPDATE jobs SET status = 'processing', worker_id = $1, started_at = now()
WHERE id = (SELECT id FROM jobs WHERE status = 'pending' ORDER BY created_at LIMIT 1 FOR UPDATE SKIP LOCKED)
RETURNING *;
```

## Data Access Patterns

### Eliminate N+1 Queries
```sql
-- BAD: N+1 — 100 users → 101 queries
SELECT id FROM users WHERE active = true; -- then for each: SELECT * FROM orders WHERE user_id = N

-- GOOD: Single query
SELECT u.id, u.name, o.* FROM users u LEFT JOIN orders o ON o.user_id = u.id WHERE u.active = true;
```

### Cursor-Based Pagination (O(1) regardless of depth)
```sql
-- BAD: OFFSET gets slower with depth
SELECT * FROM products ORDER BY id LIMIT 20 OFFSET 199980; -- Scans 200k rows!

-- GOOD: Cursor-based (always fast)
SELECT * FROM products WHERE id > 199980 ORDER BY id LIMIT 20;
```

## Review Checklist

### Query Performance (CRITICAL)
- [ ] Slow queries identified (EXPLAIN ANALYZE used)
- [ ] Missing indexes added for WHERE, JOIN, ORDER BY columns
- [ ] Composite indexes created for multi-column filters
- [ ] No sequential scans on large tables
- [ ] Full-text search uses GIN indexes, not LIKE '%...%'
- [ ] Cursor-based pagination over LIMIT/OFFSET

### Schema Design (HIGH)
- [ ] Primary keys on all tables (UUID preferred in Supabase)
- [ ] Foreign key constraints with proper ON DELETE behavior
- [ ] Appropriate data types (not all varchar/text)
- [ ] Timestamps with time zone (timestamptz)
- [ ] NOT NULL constraints where applicable

### Supabase RLS (HIGH)
- [ ] RLS enabled on all tables
- [ ] Policies use auth.uid() or auth.jwt() correctly
- [ ] No RLS bypass (service_role key usage documented)
- [ ] Policies cover ALL operations
- [ ] Policy performance checked (SELECT wrapping pattern)

### Security (CRITICAL)
- [ ] No raw SQL concatenation (parameterized queries or Supabase client)
- [ ] pgcrypto for sensitive data encryption
- [ ] Connection pooling configured (PgBouncer if Supabase)
- [ ] statement_timeout set to prevent runaway queries
- [ ] Audit logging for sensitive operations

### Migrations (MEDIUM)
- [ ] Migrations are idempotent (IF NOT EXISTS / IF EXISTS)
- [ ] Down migrations provided for rollback
- [ ] No destructive changes without data migration plan

## Common Anti-Patterns
```sql
-- BAD: Sequential scan on large table
SELECT * FROM orders WHERE status = 'pending';
-- GOOD: With index
CREATE INDEX idx_orders_status ON orders(status);

-- BAD: RLS without auth check
CREATE POLICY "all_can_read" ON users FOR SELECT USING (true);
-- GOOD: User-specific policy
CREATE POLICY "own_data" ON users FOR SELECT USING (auth.uid() = id);
```

## Approval Criteria
- **Block** — Missing RLS, SQL injection risk, sequential scans on large tables
- **Warning** — Missing indexes, nullable columns that shouldn't be, poor data types
- **Suggest** — Composite indexes, schema normalization, migration patterns
