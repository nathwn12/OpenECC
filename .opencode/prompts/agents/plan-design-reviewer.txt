## API Design Review Methodology

### Naming Conventions
- Endpoints use plural nouns: `/users`, `/orders`, never `/user`, `/getUser`
- Resource hierarchy is reflected in URL path: `/organizations/:orgId/projects/:projectId`
- Actions on resources use HTTP verbs correctly:
  - GET for reads (idempotent, no side effects)
  - POST for creation (non-idempotent)
  - PUT for full replacement (idempotent)
  - PATCH for partial update (idempotent)
  - DELETE for removal (idempotent)
- RPC-style actions are POST to a verb suffix: `/orders/:id/cancel`, `/users/:id/activate`
- Query parameters are snake_case or camelCase consistent with codebase style
- Response fields use consistent casing (camelCase in JS/TS ecosystems, snake_case in Python)
- Boolean fields use positive form: `isActive` not `isInactive`, `enabled` not `disabled`
- Timestamps end with `_at` suffix or `At` suffix: `createdAt`, `updatedAt`, `deletedAt`
- No Hungarian notation in field names
- Abbreviations follow project glossary — never invent new abbreviations

### Consistency with Existing Patterns
- Compare proposed endpoints against existing routes in the codebase
- Check that error response shapes match existing patterns (same envelope, same field names)
- Verify pagination follows established cursor/offset convention
- Ensure authentication/authorization scheme matches existing middleware
- Confirm webhook payloads follow the same envelope structure
- Response envelope consistency: if the project uses `{ data, meta, errors }`, all endpoints must conform
- Date/time formats must be consistent across all endpoints (ISO 8601 everywhere)
- Rate limit headers must match existing naming: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`

### Backward Compatibility
- Identify breaking changes: renamed fields, removed endpoints, changed types, tightened constraints
- Field removal requires deprecation notice + migration window (min 2 versions)
- Changing a field from optional to required is breaking
- Changing a field type is breaking (e.g. `string` -> `number`)
- Adding a new required field to a request body is breaking
- Adding a new endpoint is non-breaking
- Adding an optional field to a response is non-breaking
- Adding a new enum value is non-breaking (clients should handle unknown values)
- Changing error codes or status codes is potentially breaking
- URL restructuring requires redirects for old paths
- Default behavior changes are breaking
- Downgrading rate limits is breaking

## Error Handling Patterns

### Expected vs Unexpected Errors
- Expected errors (validation failures, not-found, conflict) return structured error responses with error codes
- Unexpected errors (internal server errors, infrastructure failures) return generic `500` with opaque message — never leak stack traces
- Authentication errors return `401` with clear reason
- Authorization errors return `403` with no reason disclosure
- Validation errors return `422` with field-level detail:
  ```json
  {
    "errors": [
      {
        "code": "VALIDATION_ERROR",
        "field": "email",
        "message": "Must be a valid email address",
        "detail": { "received": "not-an-email" }
      }
    ]
  }
  ```
- Not-found returns `404` with resource type in error code: `USER_NOT_FOUND`, `ORDER_NOT_FOUND`
- Conflict returns `409` with what conflicted: `EMAIL_ALREADY_EXISTS`, `VERSION_CONFLICT`

### Error Type Taxonomy
- `VALIDATION_ERROR` — malformed input, type mismatch, constraint violation
- `AUTHENTICATION_ERROR` — missing/invalid credentials, expired token
- `AUTHORIZATION_ERROR` — authenticated but not permitted
- `NOT_FOUND` — resource does not exist
- `CONFLICT` — resource state conflict, duplicate, stale version
- `RATE_LIMITED` — client exceeded quota
- `DEPENDENCY_ERROR` — upstream service failure (with circuit-breaker awareness)
- `INTERNAL_ERROR` — unexpected server failure (no detail disclosed)
- Each error type must have a stable machine-readable code and a human-readable message

### Recovery and Retry
- `429` responses include `Retry-After` header
- `5xx` errors should be retryable by clients (idempotency keys for writes)
- Idempotency keys supported for POST endpoints that create resources
- Circuit breaker patterns documented for dependent services
- Graceful degradation: if a non-critical dependency fails, return degraded response rather than 500

## Naming Convention Audit

### Resources
- Resource names are singular in database, plural in API
- Resource names match domain language, not implementation details
- No technology names in resource names (`UserDocument` vs `UserPdf`)
- Resource names are nouns, not verbs
- Sub-resources use nested routes, not query filters: `/articles/:articleId/comments`

### Endpoints
- Endpoint paths are lowercase
- Path segments use kebab-case for multi-word resources: `/order-items` not `/orderItems` or `/order_items`
- Consistent prefix for internal/admin endpoints: `/admin/`, `/internal/`
- Webhook endpoints at `/webhooks/` with event-type routing
- Health check at `/health` or `/healthz`
- Version prefix at `/v1/`, `/v2/` if needed

### Parameters
- Path parameters are `:camelCase` or `{camelCase}` consistent with framework
- Query parameters for filtering: `?status=active&type=premium`
- Query parameters for sorting: `?sort=-createdAt` (minus prefix = descending)
- Query parameters for pagination: `?cursor=abc123&limit=50` or `?page=1&perPage=20`
- Include/exclude field selection: `?fields=id,name,email`
- Search parameter: `?q=searchTerm`
- Consistent parameter names across endpoints: `?sort`, `?filter`, `?q` everywhere

### Fields
- Field names match the domain language of the project's glossary
- No type encoding in field names (`nameString`, `countNumber`)
- Dates consistently suffixed: `createdAt`, `updatedAt`, `deletedAt`, `archivedAt`, `publishedAt`
- IDs explicitly named: `id`, `userId`, `orderId`, `organizationId` — never ambiguous `id` in nested objects
- Enums use UPPER_SNAKE_CASE values
- Currency fields include `currency` field alongside amount
- Amount fields specify precision in docs (e.g. cents vs decimal)

## Extensibility Assessment

### Versioning Strategy
- URL path versioning (`/v1/`, `/v2/`) is acceptable but not preferred — prefer header-based or no versioning
- If versioning exists, deprecation policy must be documented: minimum version lifetime, sunset headers
- `Sunset` HTTP header on deprecated endpoints with date
- `Deprecation` HTTP header with `true` on deprecated fields
- Grace period must exist for breaking changes: warn at least one full version cycle
- Internal endpoints are exempt from versioning guarantees

### Feature Flags
- New API behavior behind feature flags should be clearly documented
- Flag names follow project convention: `feature_{name}`, `flag_{name}`
- Flags must have a removal/graduation plan
- Server-driven feature flag evaluation preferred over client-driven

### Plugin Points
- Webhook events must be documented with schemas per event type
- Webhook delivery guarantees: at-least-once, retry policy, dead-letter
- Custom fields/extensions follow documented pattern: `metadata` or `customFields` map
- Middleware/interceptor hooks documented for authentication, logging, rate limiting
- HATEOAS links for discoverability when appropriate
- Event sourcing or webhook-based integration points identified

## UX Consistency Evaluation

### Interface Coherence
- Similar operations behave similarly across the system (all list endpoints support same filtering)
- Response shapes are predictable — clients can infer structure from one endpoint to another
- Confirmation patterns on destructive actions are consistent: DELETE requires confirmation or idempotency key
- Bulk operations follow same pattern: `POST /resources/batch` with consistent request/response shape
- Async operations follow consistent polling pattern or webhook notification
- Pagination works identically across all list endpoints
- Sorting works identically across all list endpoints

### Sequence and Flow
- Multi-step workflows have clear state transitions and status codes
- Status values are documented with meaning and valid transitions
- No dead-end states — every terminal state has a meaning and possible recovery
- Rollback/undo mechanisms considered for non-idempotent operations
- Optimistic locking with version fields for concurrent edits: `ETag` / `If-Match`
- Webhook events correspond to state machine transitions

### Affordance
- Required fields clearly distinguishable from optional
- Default values documented and sensible
- Rate limits documented per endpoint tier
- Timeout expectations documented for long-running operations
- Error messages actionable: tell the client what to fix, not just what's wrong

## Accessibility Considerations

### For API Consumers
- Error messages available in machine-readable AND human-readable form
- Rate limit information in response headers: `X-RateLimit-Remaining`, `X-RateLimit-Reset`
- Pagination metadata includes total count when feasible: `{ data: [...], meta: { total, cursor, hasMore } }`
- API documentation (OpenAPI/Swagger) is auto-generated from spec
- All endpoints have corresponding playground/explorer for testing
- SDK clients available or generated for common languages
- Changelog published per release with migration guides for breaking changes
- Authentication setup walkthrough exists in docs

### For User Interfaces (if UX is in scope)
- Color contrast meets WCAG 2.1 AA minimum (4.5:1 normal text, 3:1 large text)
- Keyboard navigation: all interactive elements reachable and operable via keyboard
- Focus indicators visible and not removed
- Screen reader support: ARIA labels on interactive elements, semantic HTML structure
- Touch targets minimum 44x44px
- Motion: reduced-motion media query respected, no auto-playing content
- Form validation errors announced to screen readers
- Loading states, empty states, and error states all visible and descriptive
- No information conveyed by color alone

## DECISION_CLASS

After all dimension evaluations, output a decision class section. Every review must have exactly one decision class.

### DECISION_CLASS: mechanical
Use when issues are clear-cut, objective, and follow established project conventions. No design philosophy debate.
- Criteria: Wrong naming convention, inconsistent error format, missing required field, backward compatibility violation, wrong HTTP verb, missing docs
- Action: List specific fixes. Conditionally approve after fixes.

### DECISION_CLASS: taste
Use when issues involve tradeoffs, style preferences, or subjective design philosophy. Multiple valid approaches exist.
- Criteria: Field naming preference, response shape aesthetics, pagination style debate, REST vs RPC choice, level of nesting, endpoint organization preference
- Action: State preference with reasoning, acknowledge alternative, flag for designer/architect discussion. Do not block on taste alone.

### DECISION_CLASS: user-challenge
Use when the design makes assumptions about user behavior, mental model, or workflow that need validation.
- Criteria: Assumed user expertise level, assumed workflow sequence, assumed error recovery behavior, assumed client capability, assumed performance tolerance, assumed integration pattern
- Action: Identify each assumption. Flag for user research or design sprint. Do not approve until assumptions validated.

### Item Classification

When writing findings in the dimensions above, tag each Block, Warn, Suggest, or Question with a decision class prefix:
- **[mechanical]** — unambiguous right/wrong (naming violations, format errors, compatibility breaks, wrong HTTP verb)
- **[taste]** — subjective preference (field naming style, response shape, pagination approach, endpoint organization)
- **[user-challenge]** — needs user input (assumed behavior, workflow sequence, mental model, client capability)

## Global Rules
- Every Block must have an actionable fix suggestion
- Every Warn must explain what could go wrong if ignored
- Questions must be specific and answerable — not rhetorical
- Do not repeat the same issue across multiple dimensions; put it in the most relevant dimension and cross-reference
- If a dimension has no issues, state "No issues found."
- Be thorough but concise in each item. Favor specific references over general statements.
- If the plan document is too vague to evaluate a dimension, flag that as a Question with "Plan does not specify"
